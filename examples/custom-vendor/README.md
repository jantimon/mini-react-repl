# Custom vendor example

A minimal Vite app that builds its own vendor bundle (`nanoid` on top of the
React core) instead of using `mini-react-repl/vendor-default`. The
iframe-rendered REPL shows a small list whose row ids are generated by
`nanoid()`, and the seed `App.tsx` is live-editable.

## Layout

| File                                   | Role                                                                            |
| -------------------------------------- | ------------------------------------------------------------------------------- |
| `src/vendor.entry.ts`                  | manifest declaring what's in the bundle                                         |
| `package.json` `build:vendor`          | runs `repl-vendor-build src/vendor.entry.ts`                                    |
| `src/vendor.generated/index.ts`        | exports `customVendor: VendorBundle` with lazy import map + lazy types          |
| `src/vendor.generated/import-map.json` | `{ imports: { 'react': 'data:...', ... } }` — code-split, fetched on Repl mount |
| `src/vendor.generated/types.json`      | `.d.ts` payload — code-split, fetched only when the editor mounts               |
| `src/App.tsx`                          | imports `customVendor`, mounts `<Repl/>`                                        |
| `src/initial/App.tsx`                  | the seed source loaded as `?raw` text into the REPL                             |

The entire `src/vendor.generated/` folder is gitignored. A `postinstall`
script runs `build:vendor` so that fresh clones populate the folder
without a manual step; `dev` and `build` re-run it as well.

## Run

```sh
# from the repo root, build the workspace once so dist/ exists:
pnpm install
pnpm build

# then in this directory:
pnpm dev
```

`pnpm dev` runs `build:vendor` first (regenerates `src/vendor.generated/`)
and then starts Vite at http://localhost:5174.

## Lazy import map + lazy types

The generated `index.ts` consumes both `import-map.json` and `types.json`
through dynamic `import()`, so Vite/Rollup splits each payload into its
own chunk. Routes that never mount `<Repl/>` ship neither. The import-map
chunk is fetched on `<Repl/>` mount; the types chunk is fetched on
`<EditorHost/>` mount (preview-only consumers never download types).

For the host bundle this means: instead of the full ~1.5 MB import map
being inlined as a JS object literal in the entry chunk, the entry chunk
contains only a tiny thunk; the import-map fetch races the rest of the
React render work. SSR is a no-op — the thunks short-circuit when
`window` is undefined, so the chunks stay out of the SSR bundle.

## How `vendor.entry.ts` works

```ts
export * from 'mini-react-repl/vendor-base'; // 6 required-core specs

import * as nanoid from 'nanoid';

export { nanoid }; // import-map key 'nanoid'
```

Each exported namespace becomes one import-map entry; the source of the
`import * as` is what gets bundled. Aliasing is just a rename
(`export { lodashEs as 'lodash' }` would expose lodash-es as `lodash`).

## Why an entry file instead of `--packages a,b,c`?

- knip / dep-check sees real imports, no false-positive "unused" flags
- Comments live next to each entry explaining the why
- Aliasing falls out of standard ESM
- Works in monorepos with TS-source workspace packages — esbuild resolves
  them, sidestepping Node's strip-types-in-`node_modules` restriction
